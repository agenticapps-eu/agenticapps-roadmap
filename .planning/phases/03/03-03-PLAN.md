GqlResponse shape (from map.ts) — the FIXTURE CONTRACT (read carefully, this is load-bearing):
- A `GqlResponse` is the FULL envelope: `{ data: { initiatives: { nodes: [...] }, projects: { nodes: [...] } }, errors?: [...] }`.
- `mapWorkspace(json: GqlResponse)` reads `json.data.initiatives` / `json.data.projects` — i.e. it expects the top-level `data` key to already be present on its argument.
- THEREFORE the fixtures `gqlClean` and `gqlWithEmail` MUST each be a COMPLETE GqlResponse INCLUDING the top-level `data` key.
- The fetch stub's `.json()` MUST resolve to the fixture DIRECTLY — i.e. `json: async () => gqlClean` — NOT `{ data: gqlClean }`. Wrapping in `{ data: ... }` would double-nest `data` and break `mapWorkspace`. This is the contradiction the cross-AI review flagged; the chosen contract is "fixture = full GqlResponse, stub returns it directly." Do not reintroduce the wrapper.
- `data.initiatives.nodes: Array<{ id, name, color: string|null, state: string }>`
- `data.projects.nodes: Array<{ id, name, description: string|null, initiative: { id } | null, state: { name, type }, priority: number, startedAt: string|null, targetDate: string|null, projectMilestones: { nodes: Array<{ id, name, targetDate }> }, issues: { nodes: Array<{ state: { type } | null }> } }>`
- optional `errors?: Array<{ message }>`
  NOTE: existing __fixtures__/raw-clean.ts is RawWorkspace-shaped (flat `initiativeId`, no `data` envelope). The new fixtures MUST be the GQL envelope shape above (nested `initiative: { id }`, wrapped in `data`). Reuse raw-malicious.ts's email `secret@example.com` for the leak fixture.
