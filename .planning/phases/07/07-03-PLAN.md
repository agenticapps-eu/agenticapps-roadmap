<!-- Route contract from 07-02 (dispatch/status Pages Functions). No exploration needed. -->

POST /api/backfill/dispatch  body { project: string (config key), mode: "dry-run" | "apply", previewRunId?: number }
  → 200 { runId: number } | { runId: null, correlationId: string }
  → 400 (bad input / apply without previewRunId) | 403 (preview verify failed) | 5xx
GET  /api/backfill/status?run=<positiveInt>   OR   ?correlationId=<uuid>
  → 200 { status: "queued" | "in_progress" | "completed", conclusion: string | null,
          diff?: { milestones: number; issues: number; labels: number; dates: number } }
  → 400 | 403 | 5xx

DISPATCH UNION (RESOLUTION 2 — the 204 fallback MUST be handled, not a dead path):
`dispatchBackfill` resolves to `{ runId: number }` OR `{ runId: null; correlationId: string }`
OR a typed failure. `pollBackfillStatus` accepts EITHER handle and polls `?run=` or
`?correlationId=` accordingly (finding #11).

TERMINAL CONCLUSION MAP (finding MEDIUM): treat `conclusion === "success"` → success;
`conclusion === "cancelled"` → cancelled; any OTHER completed conclusion
(`failure`,`timed_out`,`startup_failure`,`stale`,`skipped`,`action_required`,`neutral`,…)
→ failure. If polling exhausts `maxTicks`, or a response is malformed after retries, or
the correlation id never resolves → `unknown` (NOT failure).

TRANSIENT vs TERMINAL (finding MEDIUM): a single 502 / network throw / malformed body /
non-completed read is a transient OBSERVATION failure — retry with backoff up to a bounded
retry count; only a completed non-success conclusion is a TERMINAL failure. This prevents a
rollback+re-enable from letting the user dispatch a second run over a live one.

Optimistic-state shape (RESEARCH "Don't Hand-Roll"; owner is OverviewPage in plan 07-04):
  Map<projectId, { pendingBackfill: boolean; planAheadOverride?: boolean }>

EXPLICIT useBackfill CONTRACT (finding MEDIUM — hook/UI interface underspecified):
  useBackfill(setBackfillState) → {
    startPreview(projectKey: string): Promise<void>;          // dispatch dry-run, poll, store diff + previewRunId
    applyBackfill(projectId: string, projectKey: string): Promise<void>; // optimistic flip → dispatch apply (with stored previewRunId) → poll → outcome
    diffFor(projectId: string): { milestones: number; issues: number; labels: number; dates: number } | undefined;
    statusFor(projectId: string): "idle" | "previewing" | "pending" | "done" | "error" | "unknown";
    errorFor(projectId: string): string | undefined;
    clearError(projectId: string): void;                      // dismiss the inline error
  }
  Argument typing is consistent: projectId (Linear id / Map key) and projectKey
  (sync.config key sent to dispatch) are BOTH strings — never an object; no dual string/object
  overloads (finding MEDIUM). The hook stores the successful dry-run's previewRunId per
  project internally so applyBackfill can pass it (dispatch enforces preview-before-apply).

Discipline to mirror — src/lib/roadmap/loader.ts lines ~34-51 (single try/catch, never
throw past the caller). vitest env is `node`; poll-loop timing is tested with
`vi.useFakeTimers()` + `vi.advanceTimersByTime()` (proven in functions/api/linear/[[path]].test.ts).
The `useBackfill` hook is thin glue; a fake-timers cleanup test asserts an
unmount/abort mid-poll issues no further fetch.
