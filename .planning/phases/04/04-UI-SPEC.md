## Color

### System Tokens (from `src/index.css`)

The app uses Tailwind v4 CSS custom-property syntax: `bg-(--color-background)`, `text-(--color-muted-foreground)`, etc. All references below follow this pattern.

| Role | Light value | Dark value | Usage |
|------|-------------|------------|-------|
| Dominant (60%) | `--background` `oklch(1 0 0)` | `oklch(0.145 0 0)` | Page background; lane row backgrounds |
| Secondary (30%) | `--muted` `oklch(0.97 0 0)` | `oklch(0.269 0 0)` | Parking rail column background; month axis row background; HoverCard/Popover content background |
| Accent (10%) | Initiative color (see below) | same | Bar fills; pill border + tint; lane color swatch; today marker; milestone markers |
| Destructive | `--destructive` `oklch(0.577 0.245 27.325)` | `oklch(0.704 0.191 22.216)` | No destructive actions in this phase — token present for future use |

Accent reserved for: bar fills, undated pill border-left stripe and background tint, initiative lane color swatch (8px × 8px circle), "today" vertical marker line, milestone diamond markers on bars. These are the only elements that receive initiative color. All other interactive states use system tokens.

### Initiative Color Ramp (D-11)

| Initiative | API Color | Rendered As |
|-----------|-----------|-------------|
| Callbot | `#0ea5e9` | sky-500 — use directly |
| agenticapps-workflow | `null` | `#10b981` (emerald-500, fallback A) |
| Factiv | `#f2994a` | orange — use directly |
| cPARX | `#5e6ad2` | linear-purple — use directly |
| fx-signals | `#f2c94c` | yellow — use directly |

Fallback algorithm (deterministic, stable across renders):

```typescript
const FALLBACK_PALETTE = ["#10b981", "#6366f1", "#14b8a6", "#ec4899", "#f97316"];

function resolveInitiativeColor(initiative: Initiative, sortedNullIds: string[]): string {
  if (initiative.color) return initiative.color;
  const idx = sortedNullIds.indexOf(initiative.id);
  return FALLBACK_PALETTE[idx % FALLBACK_PALETTE.length];
}

// sortedNullIds = initiatives
//   .filter(i => !i.color)
//   .map(i => i.id)
//   .sort()   // lexicographic — stable
```

The fallback palette colors do not clash with any existing named initiative colors.

### Bar and Pill Surface Colors

| Element | Background | Border | Text |
|---------|-----------|--------|------|
| Scheduled bar | `{initiativeColor}` at 80% opacity | none | white or `#1a1a1a` (contrast-checked per bar; see below) |
| Undated pill | `{initiativeColor}` at 8% opacity | 1.5px dashed `{initiativeColor}` | `{initiativeColor}` |
| Milestone marker | `{initiativeColor}` | white 2px | — |
| Today line | `--foreground` | — | — |
| Clamped-bar arrow | `--muted-foreground` | — | — |

Bar text contrast: use white (`#ffffff`) when the initiative color luminance < 0.4 (dark colors), else use `#1a1a1a` (near-black). The deterministic palette and named colors all qualify for white text except `#f2c94c` (yellow, luminance ≈ 0.7) — use `#1a1a1a` for fx-signals bars.

## Time Axis Contract (D-01, D-02, D-03)

### Window

- Start: first day of current month (`new Date()` → `startOfMonth`)
- End: last day of (current month + 6) — 7 months total
- Current month is July 2026 → window is 2026-07-01 … 2027-01-31
- Columns: 7 (one per calendar month)

### Today Marker (D-02)

- A 1.5px vertical line spanning the full height of the grid (axis row + all lanes)
- Color: `--foreground`
- Position: calculated as `(dayOfMonth - 1) / daysInMonth × columnWidth` within the current month column
- `aria-label="Today"` on the marker element
- Not interactive (no hover/popover)

### Off-Window Clamping (D-03)

For a scheduled project whose bar extends beyond either window edge — including projects entirely before or after the window:

| Case | Rendering |
|------|-----------|
| Bar starts before window, ends within | Clamp left edge to column 0; show `◀` (ChevronLeft 10px) at left edge of bar |
| Bar starts within, ends after window | Clamp right edge to column 6 end; show `▶` (ChevronRight 10px) at right edge |
| Bar entirely before window | 32px stub at left edge of column 0; show `◀` at left edge |
| Bar entirely after window | 32px stub at right edge of column 6; show `▶` at right edge |

"cPARX Prototype" (Apr 13–May 8 2026) falls in the "entirely before window" case — render as a 32px stub with ◀ indicator. Both arrows use `--muted-foreground` color; their `aria-label` is "Continues before window start" / "Continues past window end".

Never demote a scheduled project (has `targetDate`) to an undated pill regardless of its dates relative to the window.

## Parking Rail Contract (D-04)

Position: LEFT of the month grid. Each initiative lane contains its own parking rail section.

Pill visual:

```
┌─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┐  h=28px
  Project Name (truncated)       ← text-xs, regular, #initiativeColor
└─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┘
  border: 1.5px dashed {initiativeColor}
  bg: {initiativeColor} at 8% opacity
  border-radius: var(--radius-xl) ≈ 0.875rem
  padding: 0 8px
  width: rail width − 16px (8px margin each side)
```

Pill ordering within a lane's parking rail: sort by `priority` ascending (0=urgent first, 3=low last), then `name` ascending as tiebreaker. This surfaces the most urgent work at the top of each rail section.

Parking rail column header ("Needs dates" label): 12px, regular, `--muted-foreground`. This frames the parking rail as a call-to-action signal, not a secondary list.

## Bar Contract (D-05, D-06, D-07)

A project is "scheduled" iff `targetDate !== null` (D-06 — matches existing `TimelinePage.tsx` convention).

| Condition | Rendering |
|-----------|-----------|
| Has `startDate` AND `targetDate` | Bar from `startDate` column position → `targetDate` column position |
| Has `targetDate`, no `startDate` | 64px-wide bar ending at `targetDate` column position (D-07) |
| `targetDate === null` | Undated pill in parking rail (D-04) — never a bar |

Bar positioning (in-lane within `ScheduledGrid`):

```
left  = (daysBetween(windowStart, startDate) / windowDays) × 100%
width = (daysBetween(startDate, targetDate) / windowDays) × 100%
```

For D-07 short bars: `width = 64px` (fixed, not percentage), right-aligned to targetDate position.

Multiple bars per initiative lane stack vertically (additional rows at 48px each) if two or more scheduled projects in the same initiative overlap in time. In the current data (1 bar per initiative), no stacking occurs.

## Milestone Marker Contract (D-10)

Milestone markers appear on their parent project bar.

Visual:

- Shape: 10×10px diamond (CSS `transform: rotate(45deg)` on an 8×8px square)
- Fill: `{initiativeColor}` at 100% opacity
- Stroke: 2px solid white (provides contrast against the bar fill)
- Position: absolute within the bar, centered vertically, at the `targetDate` column position
- `title` attribute: `{milestone.name} — {milestone.targetDate}` (native tooltip as accessibility fallback)
- If the milestone's `targetDate` is null, the marker is omitted

When two or more milestones cluster within 12px of each other, collapse them to a single stacked indicator showing a count badge (e.g., "3") in `text-xs/semibold`.

## Hover/Tap Popover Contract (D-08, D-09)

### Trigger Mechanism

- Desktop (pointer: fine): `HoverCard` — opens on `mouseenter` after 300ms delay, closes on `mouseleave` after 200ms delay. No click required.
- Touch (pointer: coarse): `Popover` — opens on tap; closes on second tap or backdrop click.
- Detection: CSS `@media (pointer: coarse)` at runtime via `window.matchMedia('(pointer: coarse)').matches`. The component switches trigger mode on mount.
- Both components render the same `ProjectPopoverContent` child — no duplication of content markup.

### Popover Contents (D-09)

Width: 280px (desktop) / 260px (mobile). `max-h-[420px]` with internal `overflow-y-auto`.

```
ProjectPopoverContent
├── Header row (flex, justify-between, items-start, pb-2 border-b)
│   ├── project.name             text-sm/semibold, truncate
│   └── [status badge] [priority badge]   shadcn Badge, outline variant
│
├── Date row (flex gap-4, py-2, text-xs/regular, --muted-foreground)
│   ├── "Start: {startDate | '—'}"
│   └── "Target: {targetDate | '—'}"
│
├── Issue counts bar (py-2)
│   ├── label "Issues"  text-xs/regular, --muted-foreground
│   └── Segmented bar (h-2, rounded-full, overflow-hidden)
│       ├── Backlog segment   bg-(--color-muted)
│       ├── Started segment   bg-sky-500
│       └── Done segment      bg-emerald-500
│       (widths = count/total × 100%; all-zero → single muted bar)
│   └── count row: "24 backlog · 0 started · 0 done"  text-xs/regular
│
├── Milestones list (py-2 border-t, max 5 shown — scroll for more)
│   └── row × N: "◆ {milestone.name}  {milestone.targetDate | '—'}"
│       text-xs/regular; ◆ in {initiativeColor}
│
├── Summary (py-2 border-t, if project.summary)
│   └── text-xs/regular, text-(--color-muted-foreground), max-3-lines clamp
│
└── Footer (pt-2 border-t)
    └── <a href={project.url} target="_blank" rel="noopener noreferrer">
          Open in Linear ↗
        </a>   text-xs/semibold, text-(--color-primary)
```

If `project.url` is null/absent (D-13 pipeline not yet run): omit the footer link; the popover remains useful without it.

Priority badge labels (numeric priority → display):

| `priority` value | Display |
|-----------------|---------|
| 0 | Urgent |
| 1 | High |
| 2 | Medium |
| 3 | Low |
| other | — |

## Responsive Contract (D-12)

| Breakpoint | Behavior |
|-----------|---------|
| ≥1024px (lg) | Full layout; parking rail 160px; all bar text visible if bar ≥48px |
| 768–1023px (md) | Horizontal scroll on `main`; `min-w-[840px]` on grid; rail 160px |
| <768px (sm) | Horizontal scroll on `main`; `min-w-[840px]` on grid; rail 120px; bar text suppressed (name via popover only) |

The entire timeline grid (AxisRow + InitiativeLanes) is wrapped in `overflow-x-auto` on `main`. The AppHeader remains full-width sticky at all breakpoints.

Touch UX: pills and bars must have a minimum tap target height of 44px — achieved via the 48px lane height and a transparent click-target padding wrapper where needed.

## Dark Mode Contract (D-12)

Dark mode is detected via `prefers-color-scheme: dark` (CSS media query) AND the `.dark` class (set on `<html>`). Both mechanisms are already wired in `src/index.css`. No new tokens are needed.

Dark-mode-specific timeline behaviour:

- Bar fill: initiative color at 70% opacity (slightly less opaque — dark backgrounds make 80% feel heavier)
- Undated pill: initiative color at 12% bg opacity, border at 60% opacity (slightly more visible on dark)
- Today marker: `--foreground` (white in dark) — same token as light
- Lane separator: `--border` (already auto-swaps)
- Popover: `--popover` + `--popover-foreground` tokens (already defined for dark in `src/index.css`)

## States Contract (D-12)

### Loading State

Show skeleton swimlanes while the loader is fetching:

- Axis row: 7 muted skeleton rectangles (16×8px each), `bg-(--color-muted)` animated pulse
- Per lane: 2–3 skeleton pill shapes in rail; 1 skeleton bar shape in grid
- Use Tailwind `animate-pulse` class
- No text rendered; no copy needed

### Empty State

Triggered when `data.projects.length === 0`:

```
div.flex.flex-col.items-center.gap-4.py-24.text-center
  span.text-2xl  "📅"  (single emoji — intentionally decorative)
  p.text-sm.font-semibold  "No projects found"
  p.text-sm.text-(--color-muted-foreground)
    "Projects will appear once Linear is synced with your plans."
```

### Error State

Reuse the existing `liveUnavailable` pattern from `AppHeader.tsx` — if the loader returns an error, the `liveUnavailable` banner in the header already reads "live unavailable — showing snapshot". In the page body, render:

```
p.text-sm.text-(--color-muted-foreground).mt-8
  "Could not load timeline data. Switch to Snapshot mode above."
```

No separate error boundary component needed — the loader falls back to snapshot per existing REQ-LOADER behaviour.

### Initiative Lane — No Projects

If an initiative has no projects at all (neither scheduled nor undated): render the lane with only the lane header; both rail and grid sections show nothing. Lane height collapses to `auto` (just the header row, 32px). Do not hide the initiative.
