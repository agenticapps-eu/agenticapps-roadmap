<objective>
Close the HIGH-BLOCKING revalidation gap flagged in 05-REVIEWS.md (Codex, VERIFIED): React
Router 7 revalidates the root loader on EVERY navigation, and every Phase-5 filter toggle and
drill-down open/close is a `setSearchParams` navigation. With `roadmapLoader` on the `id:"root"`
route and NO `shouldRevalidate` (src/router.tsx:13), each filter change re-runs the loader → a
redundant `/roadmap.json` refetch in snapshot mode and REPEATED `/api/linear/snapshot` calls in
live mode — directly contradicting the snapshot-first / zero-network architecture.

Purpose: add a root `shouldRevalidate` policy so client-side-only param changes (filters,
`?project`) do NOT re-run the loader, while a `source` mode flip (snapshot↔live) still does.
This is pure logic — `shouldRevalidateRoadmap` is a function of `ShouldRevalidateFunctionArgs`
and is unit-testable directly (no render harness needed — Path B). It must land before the
05-06 UAT verifies "reload keeps filters with no refetch," so 05-06 depends on this plan.
Output: one exported pure function + its wiring on the root route + direct unit tests.
</objective>
